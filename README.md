## Author's design principles (stated explicitly)

These are not my preferences — they were stated by Hang Yu and should drive every future decision. Quoted phrases come from the original conversations.

1. **No thematic grouping of papers.** "I don't have that many papers" — chronological order within each section.
2. **Compact, minimalist paper listing**, with optional per-entry expand to show a description and related image.
3. **Traditional sub-page-style structure** — but the Research page ended up as a single page (after one round of feedback) rather than parent + sub-pages. All four research sections live on `/research/`.
4. **Restraint over flash.** "Making it fancy only serves to shame myself." No hero animations, no parallax, no accent color flourishes. Clean, professional, modern.
5. **Real gallery design** — fieldwork photos are the site's distinctive asset; no peer site surveyed has comparable content. The gallery deserves design effort even when other pages are restrained.
6. **Defer to the legacy Google Site bio.** Hang Yu spent care drafting it. The verbatim five-sentence bio from `econhangyu.com` is the homepage copy; do not rewrite it.
7. **Topic-not-region** for the sidebar bio. The earlier draft implied all work happens in Sub-Saharan Africa, which is inaccurate (recent AI/labor-market work is China-based).
8. **Downplay ISSCAD** in the homepage bio (do not insert it into the first sentence). It still appears on the Teaching page where the MPA advisees and developing-country PhD students live, which is its natural place.
9. **Selectivity in works in progress.** Only studies with AEA registry entries belong on the public site.

### Publication entry rendering

Each paper is one Markdown file in `_publications/` with **frontmatter only** (no body). The single shared template `_includes/publication-entry.html` handles all rendering. Key behaviors:

- **Title is a hyperlink** that points directly to the PDF (or to a `link:` field if there's no PDF). No separate `[PDF]` button.
- **Title is plain weight**, not bold, with no underline at rest. On hover it gets a clean dotted underline via `text-decoration-thickness: 1px; text-underline-offset: 3px`.
- **Meta line** below the title: authors · venue · year · status, separated by middle dots.
- **Journal names are bolded** in venue strings via Markdown `**...**` syntax. The include passes `venue` through `markdownify` so the asterisks render as `<strong>` tags. Example frontmatter: `venue: "**Journal of Development Economics**, 161: 103035"`.
- **Additional resources** (replication, registry, media, data, journal, technical docs) appear as **always-visible** chip-style buttons next to the meta line. Earlier iterations gated them on hover; the author asked for them to always show.
- **Inline `[more]` toggle**: if a paper has a `description:` or `image:` field, an inline `[more]` button appears at the end of the meta line. Clicking it expands a description and related image directly below the entry. Uses native HTML `<details>` — no JavaScript dependency.
- **Random author order**: the first working paper (`2026-wp-art-incentives.md`) uses the AEA's `ⓡ` symbol per the NBER WP cover page: `Hang Yu ⓡ Jared Stolove ⓡ Dean Yang ⓡ James Riddell IV ⓡ Arlete Mahumane`. This is the standard AEA randomized-author-order indicator.
- **Numbered ordered lists** (`<ol>`): peer-reviewed articles, Chinese articles, working papers, and selected works in progress all render as numbered lists. Edited volumes use an unnumbered list (`<ul class="pub-list-unnumbered">`).

### Publication frontmatter schema

```yaml
---
title: "..."                            # required; rendered as the visible title
authors: "..."                          # required; comma- or ⓡ-separated
venue: "**Journal Name**, vol(issue): pp"  # ** for bold journal name; markdownify turns ** into <strong>
year: 2026                              # required for ordering
category: published                     # see categories below
pdf: /files/publications/<slug>.pdf     # title links here
link: https://...                       # alternative link target if no PDF
journal_link: https://...               # journal landing page
replication: https://...                # Dataverse / openICPSR / etc.
media: https://...                      # VoxDev article, blog post, etc.
media_label: "VoxDev article"           # custom label for the media chip
registry: https://clinicaltrials.gov/...
registry_label: "ClinicalTrials.gov"
registry2: https://socialscienceregistry.org/...
registry2_label: "AEA registry"
data_link: https://...
tech_doc: https://...
status: "Conditionally accepted at..."  # rendered after year, italicized
sort_order: 1                           # used for WIP ordering only
description: |                          # 2–4 sentences shown in [more] expand
  ...
image: /images/papers/xxx.jpg           # optional image shown in [more] expand
---
```

**Categories** (set in `_config.yml` under `publication_category` and used by the page templates to filter):

| Category | Used for |
|---|---|
| `published` | English peer-reviewed journal articles |
| `published_cn` | Chinese-language peer-reviewed articles |
| `edited` | Edited volumes / book chapters |
| `working` | Working papers under review or circulating |
| `wip` | Selected works in progress (AEA-registered only) |

### Typography

- **Body / page content / page titles**: **Crimson Pro** (modern serif designed for screens). Loaded from Google Fonts.
- **Sidebar / masthead / nav / in-page headings / form elements**: **Inter**. Also Google Fonts.
- **Base size**: 17px on body; page content at 0.93em (~15.8px) with line-height 1.72 for paragraphs and 1.62 for list items.
- **Why this pairing**: serif body conveys academic / scholarly, sans for navigation conveys modern / clean. This is the convention used by Stanford, MIT, Harvard, and similar institutional sites.
- Both fonts loaded via a single `@import` in `assets/css/custom.css` with `display=swap`.

## Maintenance workflow

The author uses **GitHub Desktop** (not command-line Git). All routine updates flow through the visual workflow:

1. Open or create the file you want to change in any text editor (VS Code, plain text, Finder for binary drops).
2. Save.
3. Open GitHub Desktop. The changed file appears in the left panel.
4. Type a short summary at the bottom-left (e.g., `Add 2026 working paper PDF` or `Fix typo in bio`).
5. Click **Commit to main**.
6. Click **Push origin** at the top.
7. Wait 1–3 minutes. GitHub Actions rebuilds the site; the new version is live.

Verify build success in the **Actions** tab on GitHub.com. Green check = live; red X = build failure (click in for the error message).

### Adding a new publication: full example

Suppose you have a new working paper you want to add.

1. Create `_publications/2026-wp-new-paper.md`:
   ```yaml
   ---
   title: "My Compelling New Paper Title"
   authors: "Hang Yu, Coauthor One, Coauthor Two"
   venue: "Working paper"
   year: 2026
   category: working
   pdf: /files/working-papers/2026-wp-new-paper.pdf
   description: |
     Two-to-four-sentence summary that appears when a visitor clicks [more].
   image: /images/papers/new-paper.jpg
   ---
   ```
2. Drop the PDF into `files/working-papers/2026-wp-new-paper.pdf`.
3. (Optional) Drop a related fieldwork photo or chart into `images/papers/new-paper.jpg`.
4. Commit + push via GitHub Desktop.

The paper appears automatically on the Research page under "Working Papers" — no other edits needed.

### Design / layout changes

Design changes affect every visitor at once. For risky edits (CSS overhaul, new layout, theme tweaks), use a Git branch:

1. In GitHub Desktop: **Branch → New Branch**, name it `design-experiment-XYZ`.
2. Make the edits on that branch and push.
3. On GitHub.com, open a Pull Request from the branch to `main`. The Actions tab will show a preview build.
4. Merge the PR when satisfied; the branch can then be deleted.

For **content edits**, branching is overkill — commit straight to `main`.

### Emergency rollback

Git preserves full history. If a commit breaks the site:

1. On GitHub.com, find the most recent working commit in the repo's commit list.
2. In GitHub Desktop: **History → right-click the bad commit → Revert this commit**.
3. Push. The site rebuilds within a minute to the previous state.

## Dropbox + Git coexistence

The repo lives inside a Dropbox folder. To prevent Dropbox from corrupting the Git database, the `.git/` directory has the macOS extended attribute `com.dropbox.ignored = 1` set. **This must be re-applied on any new clone**:

```bash
xattr -w com.dropbox.ignored 1 ".git"
```

Verify with:
```bash
xattr -l .git
# should show: com.dropbox.ignored: 1
```

Content files (`.md`, `.html`, images, PDFs) are still synced by Dropbox, giving a free backup. Git manages version history via GitHub. This setup means Hang Yu can edit from any machine that has both Dropbox and GitHub Desktop, and the files stay in sync without Dropbox stomping on Git's internals.

## Relationship to sibling folders

The `website/` repo is **self-contained** — everything published on the site lives inside it. But the sibling folders in `_professional documents/` (`CVs/`, `papers/`, `statements/`, `photos/`, `teaching summaries/`, etc.) serve as **source-of-truth working materials**. The flow is one-way:

| Source (sibling folder) | What flows into the repo | Destination |
|---|---|---|
| `CVs/CV_eng 202604.docx` | Word-exported PDF | `files/cv.pdf` |
| `papers/1-publications/*.pdf` | Renamed copies | `files/publications/<slug>.pdf` |
| `papers/2-working papers/*.pdf` | Renamed copies | `files/working-papers/<slug>.pdf` |
| `photos/headshot/headshot_yard2.jpg` | Cropped to square + resized | `images/profile.png` |
| `photos/website pictures/*.jpg` | sips-resized to max 1200px | `images/gallery/<section>/*.jpg` |
| `teaching summaries/teaching summary.xlsx` | Hand-extracted student lists | `_pages/teaching.md` content |
| `statements/YU_2_Research Statement_202601.pdf` | Drafted descriptions | `_publications/*.md` description fields |

**Do not symlink across this boundary** — GitHub Pages builds on GitHub's servers where sibling folders don't exist. Duplication is the correct answer. It also enforces a clean "private working / public published" distinction: anything not yet copied into `website/` is not yet public.

## DNS flip (when ready)

Currently `econhangyu.com` (registered originally at Google Domains, now managed by Squarespace Domains since 2023) still points at the legacy Google Site. To migrate to GitHub Pages:

1. **Build the new site fully** (already done).
2. **Add a `CNAME` file** to the repo root containing just `econhangyu.com`. (GitHub Desktop: Repository → Open in Finder, create file, commit, push.)
3. In the **GitHub repo settings → Pages → Custom domain**, enter `econhangyu.com` and save. GitHub will check the DNS.
4. **Update DNS at Squarespace Domains** (`domains.squarespace.com`):
   - Remove the existing Google Sites CNAME records.
   - Add four `A` records for the apex domain pointing to GitHub Pages: `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`.
   - Add a `CNAME` for `www` pointing to `econhangyu.github.io`.
5. Wait ~1 hour for DNS propagation.
6. In GitHub repo settings → Pages, **enable Enforce HTTPS** once the certificate provisioning succeeds.
7. **Update `_config.yml`**: change `url:` to `https://econhangyu.com` and `author.uri:` to the same.
8. **Keep the legacy Google Site alive** (do not delete it) for ~6 months as a safety net for stragglers.
